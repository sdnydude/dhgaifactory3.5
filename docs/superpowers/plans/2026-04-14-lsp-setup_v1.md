# LSP Setup Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Configure pyright + typescript-language-server on the host so Claude Code's built-in LSP tool works against this repo, and so `npx pyright` is usable as a local quality gate.

**Architecture:** Host-only LSP servers (no container changes). `pyrightconfig.json` at repo root uses `executionEnvironments` to point the `langgraph_workflows/dhg-agents-cloud/` subtree at its existing `.venv` and the `registry/` subtree at a docker-cp-extracted site-packages snapshot under `.pyright-stubs/`. Website and legacy subtrees excluded.

**Tech Stack:** pyright (Node CLI + language server), typescript-language-server, Docker (read-only `docker cp`), existing Python 3.11 venv at `langgraph_workflows/dhg-agents-cloud/.venv`.

**Reference spec:** `docs/superpowers/specs/2026-04-14-lsp-setup-design.md`

**Execution notes:**
- This is a config-only change. Classic TDD (test ŌåÆ fail ŌåÆ implement ŌåÆ pass) doesn't fit. Verification happens by running pyright against real files and comparing findings to the measured baseline in the spec.
- No Docker container is modified. No existing venv is modified. `docker cp` is a read-only extract.
- All work happens from repo root `/home/swebber64/DHG/aifactory3.5/dhgaifactory3.5`.

### Task 2: Extract registry container site-packages snapshot

**Files:**
- Create: `.pyright-stubs/registry/` (directory populated from container)

- [ ] **Step 1: Confirm dhg-registry-api container is running**

Run:
```bash
docker ps --filter name=dhg-registry-api --format '{{.Names}}\t{{.Status}}'
```

Expected: one line showing `dhg-registry-api` with an `Up` status. If the container is not running, start it with `docker compose up -d dhg-registry-api` from the repo root ŌĆö this is a no-op if it's already up.

- [ ] **Step 2: Confirm the container's site-packages path**

Run:
```bash
docker exec dhg-registry-api python -c "import sys; print([p for p in sys.path if 'site-packages' in p][0])"
```

Expected: a path like `/usr/local/lib/python3.12/site-packages`. **Write the exact path down ŌĆö use it in the next step instead of assuming `python3.12`.**

- [ ] **Step 3: Create stubs directory and copy site-packages out**

Run (substitute the path from Step 2):
```bash
mkdir -p .pyright-stubs/registry
docker cp dhg-registry-api:/usr/local/lib/python3.12/site-packages/. .pyright-stubs/registry/
```

Expected: no output on success. `docker cp` is a read-only extract ŌĆö the container is not modified.

- [ ] **Step 4: Spot-check the snapshot**

Run:
```bash
ls .pyright-stubs/registry/ | head -20
test -d .pyright-stubs/registry/fastapi && echo "fastapi OK" || echo "MISSING fastapi"
test -d .pyright-stubs/registry/sqlalchemy && echo "sqlalchemy OK" || echo "MISSING sqlalchemy"
test -d .pyright-stubs/registry/pydantic && echo "pydantic OK" || echo "MISSING pydantic"
```

Expected: `fastapi OK`, `sqlalchemy OK`, `pydantic OK`. If any are missing, the `docker cp` path was wrong ŌĆö go back to Step 2.

### Task 4: Create `pyrightconfig.json` at repo root

**Files:**
- Create: `pyrightconfig.json`

- [ ] **Step 1: Verify the langgraph venv is live and has expected packages**

Run:
```bash
langgraph_workflows/dhg-agents-cloud/.venv/bin/python -c "import langgraph, langchain, pydantic, sqlalchemy, anthropic; print('venv OK')"
```

Expected: `venv OK`. If this fails, the venv is broken and pyright cannot use it ŌĆö stop and investigate before writing the config.

- [ ] **Step 2: Write `pyrightconfig.json`**

Create the file with this exact content:

```json
{
  "include": [
    "langgraph_workflows/dhg-agents-cloud/src",
    "langgraph_workflows/dhg-agents-cloud/tests",
    "registry"
  ],
  "exclude": [
    "**/__pycache__",
    "**/.venv",
    "**/node_modules",
    ".pyright-stubs",
    "website",
    "langgraph-dev",
    "agents",
    "docs/archive"
  ],
  "executionEnvironments": [
    {
      "root": "langgraph_workflows/dhg-agents-cloud",
      "venvPath": "langgraph_workflows/dhg-agents-cloud",
      "venv": ".venv",
      "pythonVersion": "3.11"
    },
    {
      "root": "registry",
      "extraPaths": [".pyright-stubs/registry"],
      "pythonVersion": "3.12"
    }
  ],
  "reportMissingImports": "error",
  "reportOptionalMemberAccess": "none",
  "reportGeneralTypeIssues": "error",
  "typeCheckingMode": "basic"
}
```

- [ ] **Step 3: Validate the JSON parses**

Run:
```bash
jq . pyrightconfig.json > /dev/null && echo "valid"
```

Expected: `valid`.

- [ ] **Step 4: Run pyright with the config, no target ŌĆö full project scan**

Run:
```bash
time pyright --outputjson 2>/dev/null | jq '.summary'
```

Expected: a summary block with `filesAnalyzed` in the low-to-mid hundreds, `errorCount` somewhere in the tens-to-low-hundreds range (don't fixate on the exact number ŌĆö the goal is that it RAN and produced structured output). `time` should report under ~30s for the first run.

If pyright reports a config error (e.g., `venvPath is not valid`, `unknown executionEnvironment key`), fix the config and re-run before moving on. Do not proceed with a broken config.

### Task 5: Verify against the noise-floor baseline from the spec

**Files:** none (verification only)

- [ ] **Step 1: Check the orchestrator surfaces the real AsyncPostgresSaver bug**

Run:
```bash
pyright --outputjson langgraph_workflows/dhg-agents-cloud/src/orchestrator.py 2>/dev/null \
  | jq '.generalDiagnostics | map(select(.range.start.line >= 1793 and .range.start.line <= 1797))'
```

Expected: at least one diagnostic referencing `AsyncPostgresSaver.from_conn_string` or the `await` expression at/near line 1795. This is the real latent bug documented in the spec ŌĆö it must survive into the live config. If the array is empty, the config is not seeing the langgraph venv correctly ŌĆö investigate `executionEnvironments[0]` before proceeding.

- [ ] **Step 2: Check the orchestrator runs in under 3 seconds**

Run:
```bash
time pyright --outputjson langgraph_workflows/dhg-agents-cloud/src/orchestrator.py 2>/dev/null | jq '.summary.timeInSec'
```

Expected: `timeInSec` under 3.0, and `time` reporting a similar wall-clock. If it's significantly slower, pyright may be re-analyzing transitive dependencies ŌĆö acceptable on a cold run but worth noting.

- [ ] **Step 3: Check registry/api.py surfaces real findings**

Run:
```bash
pyright --outputjson registry/api.py 2>/dev/null \
  | jq '{summary: .summary, byRule: (.generalDiagnostics | group_by(.rule) | map({rule: .[0].rule, count: length}) | sort_by(-.count))}'
```

Expected: a non-empty `byRule` array. The spec measured 4 findings with imports disabled (Media.status assignment, client_id None-safety, two kwargs-don't-exist). With real stubs, the exact count may shift, but rules like `reportAttributeAccessIssue`, `reportArgumentType`, or `reportCallIssue` should appear. If the result is `errorCount: 0` or is dominated by `reportMissingImports`, the `extraPaths` to `.pyright-stubs/registry` is not wired correctly ŌĆö revisit Task 2 and Task 4.

- [ ] **Step 4: Confirm Optional noise is muted**

Run:
```bash
pyright --outputjson langgraph_workflows/dhg-agents-cloud/src/orchestrator.py 2>/dev/null \
  | jq '[.generalDiagnostics[] | select(.rule == "reportOptionalMemberAccess")] | length'
```

Expected: `0`. The spec's noise-floor measurement saw 29 of these before muting ŌĆö they must be suppressed now.

### Task 6: Verify Claude Code's LSP tool actually works against the repo

**Files:** none (interactive verification)

This task is interactive ŌĆö it must be run by Claude Code in the main session, not by a subagent. The LSP tool is only available in the main session.

- [ ] **Step 1: Invoke `LSP goToDefinition` on a registry symbol**

In the main Claude Code session, use the `LSP` tool with operation `goToDefinition` on a `fastapi` symbol in `registry/api.py` (e.g., the `APIRouter` import or any `FastAPI()` call).

Expected: the tool returns a location inside `.pyright-stubs/registry/fastapi/` or similar. If it returns "no definition" or errors out, pyright-langserver is not seeing the stubs ŌĆö re-check the `extraPaths` config.

- [ ] **Step 2: Invoke `LSP goToDefinition` on a langgraph symbol**

In the main session, use the `LSP` tool with operation `goToDefinition` on a symbol imported from `langgraph` in `langgraph_workflows/dhg-agents-cloud/src/orchestrator.py` (e.g., `StateGraph`, `AsyncPostgresSaver`).

Expected: the tool returns a location inside `langgraph_workflows/dhg-agents-cloud/.venv/lib/python3.11/site-packages/langgraph/`. If it returns "no definition," the `executionEnvironments[0].venvPath` is wrong.

- [ ] **Step 3: Invoke `LSP findReferences` on a project-internal symbol**

Pick a well-known symbol in the repo (e.g., the `get_db` function in `registry/api.py` or the `orchestrator_graph` in `orchestrator.py`) and call `LSP findReferences`.

Expected: the tool returns one or more locations across project files (not just the definition site). This confirms pyright-langserver can index project code, not just dependencies.

- [ ] **Step 4: Invoke `LSP goToDefinition` on a frontend symbol**

In `frontend/src/lib/inboxApi.ts` (or any TS file), pick an imported symbol (e.g., something from `@/stores/review-store`) and call `LSP goToDefinition`.

Expected: the tool returns a location inside `frontend/src/stores/review-store.ts`. This confirms typescript-language-server is also working (it uses the existing `frontend/tsconfig.json`, no new config required).
