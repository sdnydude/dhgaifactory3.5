# CME Intake Templates & Market Intelligence — Design Spec

**Date:** 2026-03-12
**Author:** Stephen Webber + Claude
**Status:** Draft — pending review
**Branch:** feature/langgraph-migration

---

## 1. Problem Statement

Every CME grant project starts from a blank 47-field intake form. This wastes time, introduces inconsistency, and ignores institutional knowledge from previously successful projects. Meanwhile, grant opportunities go undiscovered because no one is systematically monitoring pharma transparency reports, FDA decisions, RFPs, and competitive CME activity.

## 2. Solution Overview

Two connected systems:

1. **Intake Templates** — versioned, categorized prefilled intake forms that reduce the 47-field blank form to a review-and-customize experience. Created manually, saved from completed projects, or auto-generated by the intelligence agent. Refined over time based on project outcomes.

2. **CME Market Intelligence Agent** — an autonomous LangGraph agent that monitors external data sources 24/7 and recommends grant opportunities with prefilled templates. Delivers recommendations to an in-app inbox with optional email digest.

The template system is foundational and ships first. The intelligence agent depends on it.

## 3. Design Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| D1 | Templates organized by therapeutic area + disease state | Disease state is where real prefill value lives — "Oncology" is too broad, "Oncology > NSCLC" prefills 30+ fields |
| D2 | Versioned and immutable templates (Approach B) | Templates evolve from data — need audit trail, version tracking, and the ability to know which version a project used |
| D3 | "Save as Template" from completed projects + manual editor | Dual creation path covers learning from experience and manual curation |
| D4 | All fields editable with "changed from template" indicators | Full flexibility with change tracking to learn which defaults actually work |
| D5 | Refinement via post-project suggestions + aggregate drift detection | Both immediate feedback (after each project) and statistical analysis (across projects) |
| D6 | Human approval required for all template changes | Templates are institutional knowledge — no silent mutations |
| D7 | Intelligence agent creates templates with source: 'agent_recommended' | Clear provenance — you always know if a human or agent created a template |
| D8 | Inbox as primary delivery, email as opt-in | In-app is the workflow; email ensures nothing is missed |
| D9 | 7-year ACCME retention on templates linked to funded grants | Template version becomes a compliance artifact when a grant is funded from it |
| D10 | Intelligence agent built as a LangGraph graph | Fits existing infrastructure — no new services to manage |

## 4. Database Schema

### 4.1 cme_template_categories

Hierarchical taxonomy for organizing templates.

```sql
CREATE TABLE cme_template_categories (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    parent_id UUID REFERENCES cme_template_categories(id),
    name VARCHAR(200) NOT NULL,
    slug VARCHAR(100) NOT NULL UNIQUE,
    description TEXT,  -- helps users browse unfamiliar therapeutic areas
    level INT NOT NULL DEFAULT 0,  -- 0=therapeutic area, 1=disease state, 2=variant
    sort_order INT NOT NULL DEFAULT 0,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_template_categories_parent ON cme_template_categories(parent_id);
CREATE INDEX idx_template_categories_level ON cme_template_categories(level);
```

### 4.2 cme_templates

Versioned, immutable template definitions.

```sql
CREATE TABLE cme_templates (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    category_id UUID NOT NULL REFERENCES cme_template_categories(id),
    template_lineage_id UUID NOT NULL,  -- groups all versions of the same template
    version INT NOT NULL DEFAULT 1,
    is_current BOOLEAN NOT NULL DEFAULT TRUE,

    -- Identity
    name VARCHAR(300) NOT NULL,
    description TEXT,

    -- Payload
    intake_defaults JSONB NOT NULL,  -- full IntakeSubmission object

    -- Provenance
    source VARCHAR(50) NOT NULL,  -- 'manual', 'from_project', 'agent_recommended'
    source_project_id UUID REFERENCES cme_projects(id),
    created_by VARCHAR(255) NOT NULL DEFAULT 'system',

    -- Quality signals
    usage_count INT NOT NULL DEFAULT 0,
    success_count INT NOT NULL DEFAULT 0,  -- projects funded from this template
    avg_quality_score FLOAT,  -- average prose quality score across projects

    -- Compliance
    retention_until TIMESTAMPTZ,  -- set when linked to a funded project
    is_archived BOOLEAN NOT NULL DEFAULT FALSE,

    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),  -- tracks usage_count/success_count changes

    UNIQUE(template_lineage_id, version)
);

CREATE INDEX idx_templates_category ON cme_templates(category_id);
CREATE INDEX idx_templates_lineage ON cme_templates(template_lineage_id);
CREATE INDEX idx_templates_current ON cme_templates(is_current) WHERE is_current = TRUE;
CREATE INDEX idx_templates_source ON cme_templates(source);
CREATE INDEX idx_templates_source_project ON cme_templates(source_project_id) WHERE source_project_id IS NOT NULL;
```

### 4.3 cme_template_proposals

Refinement suggestions with evidence.

```sql
CREATE TABLE cme_template_proposals (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    template_lineage_id UUID NOT NULL,  -- references the lineage, not a specific version
    proposed_changes JSONB NOT NULL,  -- {field_path: {from: old, to: new}}
    proposal_type VARCHAR(50) NOT NULL,  -- 'post_project', 'aggregate_drift'
    evidence_project_ids UUID[] NOT NULL DEFAULT '{}',  -- intentional denormalization for query simplicity
    evidence_summary TEXT NOT NULL,
    status VARCHAR(20) NOT NULL DEFAULT 'pending',  -- 'pending', 'approved', 'rejected'
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    resolved_at TIMESTAMPTZ,
    resolved_by VARCHAR(255)
);

CREATE INDEX idx_proposals_lineage ON cme_template_proposals(template_lineage_id);
CREATE INDEX idx_proposals_status ON cme_template_proposals(status);
```

**Version creation semantics:** Both `PUT /api/cme/templates/{id}` and proposal approval use the same internal function `_create_template_version()` which:
1. Finds the current `is_current=true` row for the target `template_lineage_id`
2. Sets `is_current=false` on the old row
3. Creates a new row with the same `template_lineage_id`, `version` incremented by 1, `is_current=true`
4. Copies `usage_count`, `success_count`, `avg_quality_score` from the old row

Proposals reference `template_lineage_id` (not a specific version ID) so they remain valid even if new versions are created between proposal creation and approval. On approval, the new version is always created from the current `is_current=true` version.

### 4.4 cme_opportunities

Market intelligence discoveries.

```sql
CREATE TABLE cme_opportunities (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    template_id UUID REFERENCES cme_templates(id),

    -- Source
    source_type VARCHAR(50) NOT NULL,  -- 'rfp', 'fda_approval', 'transparency_report', 'competitor_activity'
    source_url TEXT,
    raw_data JSONB,

    -- Opportunity details
    title TEXT NOT NULL,
    summary TEXT NOT NULL,
    supporter_name VARCHAR(300),
    estimated_amount FLOAT,
    deadline DATE,
    priority VARCHAR(20) NOT NULL DEFAULT 'medium',  -- 'high', 'medium', 'low'

    -- Workflow
    status VARCHAR(20) NOT NULL DEFAULT 'new',  -- 'new', 'reviewed', 'pursuing', 'snoozed', 'dismissed'
    snoozed_until DATE,  -- for snooze functionality
    project_id UUID REFERENCES cme_projects(id),  -- linked when "Pursue" is clicked

    -- Audit
    created_by VARCHAR(255) NOT NULL DEFAULT 'system',  -- user email or agent identifier

    -- Compliance
    retention_until TIMESTAMPTZ,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_opportunities_status ON cme_opportunities(status);
CREATE INDEX idx_opportunities_priority ON cme_opportunities(priority);
CREATE INDEX idx_opportunities_deadline ON cme_opportunities(deadline);
CREATE INDEX idx_opportunities_supporter ON cme_opportunities(supporter_name);
CREATE INDEX idx_opportunities_template ON cme_opportunities(template_id) WHERE template_id IS NOT NULL;
CREATE INDEX idx_opportunities_snoozed ON cme_opportunities(snoozed_until) WHERE status = 'snoozed';
```

### 4.5 cme_projects additions

ALTER TABLE cme_projects

## 5. API Endpoints

### 5.1 Template CRUD

| Method | Endpoint | Purpose |
|--------|----------|---------|
| GET | `/api/cme/templates` | List templates. Filters: `?category_id=`, `?search=`, `?source=`, `?current_only=true` |
| GET | `/api/cme/templates/{id}` | Get a specific template version |
| POST | `/api/cme/templates` | Create a new template |
| PUT | `/api/cme/templates/{id}` | Create a new version (old stays, `is_current` flips) |
| DELETE | `/api/cme/templates/{id}` | Soft archive (`is_current = false`, `is_archived = true`) |

### 5.2 Template Categories

| Method | Endpoint | Purpose |
|--------|----------|---------|
| GET | `/api/cme/template-categories` | Full category tree (nested response) |
| POST | `/api/cme/template-categories` | Create a category node |
| PUT | `/api/cme/template-categories/{id}` | Update name, description, parent, sort order |
| DELETE | `/api/cme/template-categories/{id}` | Soft delete (fails if templates still reference it) |

### 5.3 Save from Project

| Method | Endpoint | Purpose |
|--------|----------|---------|
| POST | `/api/cme/projects/{id}/save-as-template` | Create template from completed project. Body: `{name, category_id, description}` |

### 5.4 Template Refinement

| Method | Endpoint | Purpose |
|--------|----------|---------|
| GET | `/api/cme/templates/{id}/proposals` | List pending proposals |
| POST | `/api/cme/templates/{id}/proposals` | Create a proposal (system-generated) |
| POST | `/api/cme/templates/{id}/proposals/{pid}/approve` | Approve → creates new template version |
| POST | `/api/cme/templates/{id}/proposals/{pid}/reject` | Dismiss proposal |

### 5.5 Opportunities

| Method | Endpoint | Purpose |
|--------|----------|---------|
| GET | `/api/cme/opportunities` | List opportunities. Filters: `?status=`, `?priority=`, `?source_type=`, `?template_id=`. Snoozed items with `snoozed_until < NOW()` are returned as status `new`. |
| GET | `/api/cme/opportunities/{id}` | Get opportunity details |
| POST | `/api/cme/opportunities` | Create an opportunity (used by intelligence agent). Body: `OpportunityCreate` model |
| POST | `/api/cme/opportunities/{id}/pursue` | Create project from opportunity's template, link them |
| POST | `/api/cme/opportunities/{id}/dismiss` | Mark as dismissed |
| POST | `/api/cme/opportunities/{id}/snooze` | Snooze for N days. Body: `{days: 30}` |

### 5.6 Using Templates in Project Creation

The existing `POST /api/cme/projects` gains an optional `template_id` field in the `IntakeSubmission` body. The backend:
1. Records `template_id` and `template_version` on the project
2. Diffs the submitted intake against `intake_defaults` to compute `template_overrides`
3. Increments `usage_count` via atomic SQL: `SET usage_count = usage_count + 1` (not ORM read-modify-write, to avoid concurrency issues)

### 5.7 User Settings

| Method | Endpoint | Purpose |
|--------|----------|---------|
| GET | `/api/cme/settings` | Get current user settings |
| PUT | `/api/cme/settings` | Update settings (email digest toggle, frequency) |

## 6. Frontend Changes

### 6.1 Template Picker on `/projects/new`

Before the intake form, a selection step:

- **"Start from template"** or **"Start blank"** — two prominent buttons
- "Start from template" opens a two-column picker:
  - Left: therapeutic areas (from `cme_template_categories` level 0)
  - Right: templates within the selected area, showing name, description, usage count, success rate badge
  - Click a template → intake form loads prefilled
  - Badge at top of form: "Based on: {template name} v{version}"
- "Start blank" → current behavior

**Quick-start mode:** When loading from a template, sections that are fully prefilled start collapsed. Only sections needing user input are expanded. Section B (Supporter Info) is always expanded first — the supporter is the variable that changes most.

### 6.2 "Changed from Template" Indicators

- Fields that differ from the template default show a small colored dot on their label
- Summary line at the top of the form: "4 of 47 fields customized from template"
- Hovering the indicator shows the original template value in a tooltip

### 6.3 "Save as Template" on Completed Projects

- Button appears on project detail page when status is `complete`
- Opens a dialog: name, category picker (dropdown tree), description
- Pre-populates intake data from the project
- Creates template with `source: 'from_project'`

### 6.4 Template Management Page (`/templates`)

New sidebar nav item. Page layout:

- Category filter sidebar (therapeutic areas as expandable groups)
- Grid of template cards showing: name, category breadcrumb, version, usage count, success rate, source badge (manual / from project / agent recommended)
- Click card → template editor (reuses the intake form layout, but editing defaults)
- "Proposals" tab on each template showing pending refinement suggestions with approve/reject buttons and evidence summary

### 6.5 Opportunity Inbox Enhancement

The existing `/inbox` page gains a second tab or section: **"Recommended Opportunities"**

Each opportunity card shows:
- Title, supporter name, estimated amount, deadline
- Priority badge (high/medium/low)
- Source badge (RFP, FDA approval, transparency report, competitor activity)
- Agent-generated summary
- Three actions: **Pursue** (creates project from template), **Snooze** (30/60/90 day picker), **Dismiss**
- Future: **Forward to team** action

### 6.6 Settings Page (`/settings`)

New sidebar nav item. Minimal v1:

- **Email digest toggle** — on/off
- **Frequency** — daily or weekly (radio/select)
- **Email address** — text input
- Save button

## 7. Template Refinement Engine

### 7.1 Post-Project Suggestions (Immediate)

Triggered when a project reaches `complete` status and has a `template_id`. The trigger fires inside `_sync_project_from_thread()` in `cme_endpoints.py`, which already updates project status from LangGraph Cloud state — the refinement check hooks into this same flow:

1. Diff the project's final intake against the template's `intake_defaults`
2. If 3+ fields differ, create a `cme_template_proposal` with:
   - `proposal_type: 'post_project'`
   - Changed fields and their new values
   - Source project ID as evidence
   - Summary: "Project '{name}' changed {N} fields from template defaults"
3. Proposal appears in the template management UI

### 7.2 Aggregate Drift Detection (Periodic)

Runs as a scheduled task (initially manual trigger, later cron):

1. For each template with 3+ completed projects since last analysis:
   - Compare all projects' intakes against the template defaults
   - Identify fields changed in >60% of projects
   - Weight successful projects (grant funded) more heavily than others
2. If systematic drift is found, create a single `cme_template_proposal` with:
   - `proposal_type: 'aggregate_drift'`
   - Statistically dominant values for each drifted field
   - All contributing project IDs as evidence
   - Summary: "Field '{X}' changed in 4 of 5 projects (3 funded)"
3. Proposal appears in the template management UI and optionally in the inbox

### 7.3 Quality Signal Tracking

On project completion:
- If project has a `template_id`, update the template's `avg_quality_score` (rolling average from prose quality agent scores)
- If project's `is_funded` is set to `true`, atomically increment `success_count` via SQL `SET success_count = success_count + 1`
- Surface success rate as a badge: `success_count / usage_count`

## 8. Market Intelligence Agent (Interface Contract)

The agent is a separate LangGraph graph designed in a follow-up spec. This section defines only the interface.

### 8.1 How the Agent Creates Recommendations

1. Agent discovers an opportunity from an external source
2. Agent calls `POST /api/cme/templates` with `source: 'agent_recommended'` to create a prefilled template
3. Agent calls `POST /api/cme/opportunities` to create the opportunity record linked to the template
4. Opportunity appears in the user's inbox

### 8.2 Agent Data Sources (v1 priorities)

| Priority | Source | Signal |
|----------|--------|--------|
| 1 | Pharma transparency/giving reports | Structured, public, annual — highest signal-to-noise |
| 2 | FDA approval calendar and decisions | New approvals create CME demand 3-6 months later |
| 3 | CME RFPs | Time-sensitive, direct opportunities |
| 4 | Competitor CME activities | Market awareness, gap identification |

### 8.3 Agent Data Sources (v2)

| Priority | Source | Signal |
|----------|--------|--------|
| 5 | Medical conferences and events | Topic trends, networking opportunities |
| 6 | Technology/AI/device approvals | Emerging CME topics |
| 7 | Clinical trial results | Future CME demand signals |
| 8 | Pharma pipeline data | Upcoming approvals = future grant opportunities |

### 8.4 Agent Design Principles

- Runs as a LangGraph graph in the existing infrastructure
- Autonomous operation — runs on schedule, not on-demand
- Creates opportunities with priority assessment (high/medium/low)
- Learns from dismiss/pursue patterns over time (future: feedback loop)

## 11. Implementation Notes

### 11.1 Authentication Model (v1)

The current system has no authentication middleware. For v1, user identity is passed as a query parameter or header (`X-User-Email`). This applies to:
- `GET/PUT /api/cme/settings` — identifies which user's settings to read/write
- `resolved_by` on proposals — set from the request header
- `created_by` on templates and opportunities — set from the request header or defaults to `'system'`

Full authentication (OAuth/JWT) is a separate concern and should not block this feature.

### 11.2 File Organization

The existing `cme_endpoints.py` is already ~2200 lines. New endpoints should be split into separate files:
- `registry/cme_template_endpoints.py` — template CRUD, categories, proposals
- `registry/cme_opportunity_endpoints.py` — opportunity CRUD, pursue/dismiss/snooze
- `registry/cme_settings_endpoints.py` — user settings

All use the same `router = APIRouter(prefix="/api/cme")` pattern and are registered in `api.py`.

### 11.3 Frontend Intake Form Refactor

The current intake form renders one active section at a time (`activeSection` state). Quick-start mode requires changing this to show all sections simultaneously with collapse/expand, which is a meaningful refactor of `intake-form.tsx`. The section components themselves (`section-a-basics.tsx`, etc.) do not need to change — only the container that renders them.

### 11.4 Snooze Resurfacing

Snoozed opportunities resurface automatically via query logic, not a background job. The `GET /api/cme/opportunities` endpoint treats snoozed items with `snoozed_until < NOW()` as status `new` in its response. No cron job needed.

### 11.5 Sidebar Updates

Two new entries needed in `frontend/src/components/layout/sidebar.tsx`:
- `/templates` (icon: `LayoutTemplate` from lucide-react)
- `/settings` (icon: `Settings` from lucide-react)
